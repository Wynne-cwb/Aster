## Critical Issues

### CR-01: AutoInsertEffect 的 hasTriggeredRef 每次 render 重建，无法防重复触发

**File:** `src/components/ChatBubble.tsx:150`

**Issue:** `hasTriggeredRef` 被声明为普通对象字面量 `{ current: false }`，而非 `useRef()` 的返回值。React 每次渲染都会重新执行函数体，因此每次 `AutoInsertEffect` 重渲染（例如父组件 `ChatBubble` 因任何 store 变化而 re-render）都会将 `hasTriggeredRef.current` 重置回 `false`。`useEffect` 内的 guard 因此形同虚设——只要组件重渲染，就可能再次调用 `acceptToolCall`，导致同一 tool call 向文档写入多次。在 auto 模式下，这是一个静默的数据写入重复缺陷。

```tsx
// 当前（有缺陷）：
const hasTriggeredRef = { current: false };  // 每次 render 重置为 false

// 修复：使用 useRef 持久化跨 render 的标志
const hasTriggeredRef = useRef(false);
```

此外，`acceptToolCall` 本身也缺少"已是 accepted 则跳过"的幂等检查（`chat.ts:269-295`）：它在写入成功后才把 status 改成 `accepted`，若 `AutoInsertEffect` 在写入完成之前又被触发，会再次调用 `adapter.insert`。建议在 `acceptToolCall` 入口增加：

```ts
if (tc.status === 'accepted') return;  // 已写入则跳过
```

### CR-02: chat.ts sendMessage 把 AsterError.message 原样写入 errorCode 读取路径，且 error.message 本身未过 sanitize 就显示到 UI

**File:** `src/store/chat.ts:232`

**Issue:** 错误处理分支把 `(e as Error).message` 直接赋给 `message.content`（即气泡文本），该内容最终显示在 `ErrorBubble` 中。如果某个异常的 `.message` 包含 API Key 片段（例如：网络层、Office.js 层或第三方 Provider 抛出含凭证的 TypeError），Key 就会渲染到 UI。

虽然 `AsterError` 子类的 message 字段自身由 `mapHttpError` 等受控函数构建、不含 Key，但外部抛出的非 `AsterError`（尤其是 Office.js `HostApiError` 经 catch 后再包装成普通 Error 的情景，或 `ProviderRegistry.resolve` 失败时抛出的 `Error('默认 Provider 未配置')`）不受此约束。一旦 `classifyFetchThrow` 的兜底分支被绕过，便存在泄露窗口。

修复策略：在赋值到 `message.content` 前对 message 字符串执行 `sanitizeErrBody`，或至少对已知安全的 `AsterError` 子类使用，其余统一用固定友好提示：

```ts
// chat.ts 232 附近
const safeMsg = e instanceof AsterError
  ? (e as AsterError).message
  : '请求遇到未知错误，请重试';

set((s) => ({
  messages: s.messages.map((m) =>
    m.id === assistantMsg.id
      ? { ...m, role: 'error', content: safeMsg, ... }
      : m,
  ),
}));
```

### CR-03: mapHttpError 未处理 403 Forbidden，回退为 NetworkError 而非 KeyInvalidError，导致用户拿到错误操作指引

**File:** `src/lib/sse.ts:109`

**Issue:** OpenAI-compatible API（DeepSeek、AiHubMix 均有记录）在 API Key 权限不足或被吊销时可能返回 `403 Forbidden`。`mapHttpError` 的 `switch` 只处理 `401` 为 `KeyInvalidError`，`403` 落入 `default` 分支，返回 `NetworkError('网络错误，请检查连接')`。

用户看到的错误 CTA 是"检查网络连接"而非"更新 API Key"，会错误引导用户排查网络而非凭证。

```ts
// sse.ts switch 补充：
case 403:
  return new KeyInvalidError('API Key 权限不足或已被吊销，请前往设置更新 Key');
```

### CR-04: chat.ts tool_call 参数校验只检查 position 枚举，未验证 text 字段是否为字符串，恶意 AI 可注入非字符串值写入 adapter

**File:** `src/store/chat.ts:182-193`

**Issue:** `JSON.parse(event.arguments)` 后只校验了 `parsedArgs.position` 是否合法枚举。`parsedArgs.text` 完全没有检查——若为 `null`、数字、或长达数 MB 的超长字符串，`adapter.insert({ type: 'text', value: tc.arguments.text, ... })` 会将非字符串值传给宿主 API（Office.js 对 `range.values` 传入 `null` 的行为是 undefined），或触发超大写入把用户文档占满。

```ts
// 在 position 校验之后补充：
if (typeof parsedArgs.text !== 'string') continue;
if (parsedArgs.text.length > 50_000) continue;  // 防止超大写入
```

## Warnings

### WR-01: PptAdapter.insert append_end 分支有三次 context.sync，违反 NFR-02 two-sync 规则

**File:** `src/adapters/PptAdapter.ts:126-133`

**Issue:** `append_end` 分支先 `shapes.load + await ctx.sync()`（sync 1），再 `tr.load('text') + await ctx.sync()`（sync 2），最后写入 + `await ctx.sync()`（sync 3）。文件注释声明只做两次 sync；NFR-02 two-sync 规则是「load → sync → write → sync」，`append_end` 多出一次 sync。三次 sync 在 Web 上每次往返约 50-100ms，累积延迟明显，且若 slide 在两次 sync 之间被外部修改，会引入竞态。

修复：在第一次 sync 时同时 load address + text，避免第二次 sync：

```ts
case 'append_end': {
  const tr = shapes.items[0].textFrame.textRange;
  tr.load('text');
  await ctx.sync();  // sync 1: load text
  tr.text = ((tr.text as string) ?? '') + content.value;
  await ctx.sync();  // sync 2: write
  break;
}
```

注意 `shapes.load + sync` 仍然需要保留（第一次），但应把 `tr.load('text')` 移入同一批。最终 sync 次数可降至 2。

### WR-02: hydrateFromStorage 覆盖 stored providers 时不合并 BUILT_IN_PROVIDERS，若用户之前删除了内置 Provider 记录，内置 Provider 无法恢复

**File:** `src/store/providers.ts:176-182`

**Issue:** 当 `stored` 非空时，`useProviderStore.setState({ providers: stored, ... })` 完全替换 providers 数组，不再包含 `BUILT_IN_PROVIDERS`。如果 `stored` 由于某种原因（localStorage 污染、迁移 bug）不含 deepseek/aihubmix 记录，内置 Provider 在重启后消失，用户无法选择内置 Provider。此外 `setSupportsToolCall` 会把 `supportsToolCall` 字段写入 `stored` providers 并持久化，若 Provider 列表从 stored 直接恢复，`isBuiltIn` 字段取决于 stored 数据中的值，可能被外部篡改（恶意的 localStorage 修改）。

建议：hydrate 时强制将 `BUILT_IN_PROVIDERS` 合并到 stored 中（以 id 为主键 merge，优先保留内置字段的 isBuiltIn=true）。

### WR-04: SelectionPill 的 onSelectionChanged 回调内 getSelection 异步调用无错误处理，HostApiError 会成为未处理 Promise rejection

**File:** `src/components/SelectionPill.tsx:40-43`

**Issue:** `adapter.onSelectionChanged` 回调是同步函数，内部调用 `adapter.getSelection()` 返回 Promise 但没有 `.catch()`。若 `getSelection()` 抛出 `HostApiError`（例如 Task Pane 切换宿主时 Office.js 上下文销毁），会产生 unhandled promise rejection，可能在 Office WebView 中触发未定义行为。

```ts
const unsub = adapter.onSelectionChanged(async () => {
  try {
    const sel = await adapter.getSelection();
    setCtx(formatSelection(sel, i18n));
  } catch {
    // 宿主 API 失败时保持上一次 ctx，不 crash
  }
});
```

### WR-05: classifyFetchThrow 的 CORS preflight 拒绝场景下 navigator.onLine 可能为 true，导致 CORS 错误被误判为 KEY_INVALID

**File:** `src/lib/sse.ts:157-185`

**Issue:** CORS preflight 被服务器拒绝（例如 Provider 的 CORS 策略不允许从 Office for Web 的 origin 访问）时，Chrome/Edge 抛出 `TypeError: Failed to fetch`，且此时 `navigator.onLine === true`、URL 是 `https://`，三条信号全满足，触发 `KeyInvalidError`。用户看到"Key 可能无效"，但实际问题是 CORS 配置，与 Key 无关，错误引导用户去更换 Key。

这是一个难以完美区分的边界情形（注释已提及"无后台无法 ping"），但错误措辞应更保守：

```ts
return new KeyInvalidError(
  'Key 可能无效或 Provider 不允许浏览器直连，请检查 API Key 与 Base URL'
);
```

### WR-06: ExcelAdapter.insert append_end 分支使用 getUsedRange(true) 后的 rowCount，若表格空则 rowCount 为 0，导致写入 A1 而非追加新行；且与 cursor/replace_selection 分支行为不一致

**File:** `src/adapters/ExcelAdapter.ts:121-130`

**Issue:** `getUsedRange(true)` 在空工作表时会抛出 `ItemNotFound` 异常（参数 `true` 表示"只返回有值的区域，若无则抛错"），而非返回 rowCount=0。当前代码 `used.rowCount ?? 0` 无法捕获这个异常，会让外层 `catch (err)` 将其包成 `HostApiError('Excel text 写回失败')`，不如预期地在 A1 写入。

修复：使用 `getUsedRange(false)` 或 catch ItemNotFound 后 fallback 到 A1：

```ts
case 'append_end': {
  try {
    const used = ctx.workbook.worksheets
      .getActiveWorksheet().getUsedRange(false);
    used.load('rowCount');
    await ctx.sync();
    const newRow = (used.rowCount ?? 0) + 1;
    ...
  } catch {
    // 空表 fallback
    const target = ctx.workbook.worksheets.getActiveWorksheet().getRange('A1');
    target.values = [[content.value]];
    await ctx.sync();
  }
}
```

### WR-07: SettingsPanel.handleSave 在 creating 模式下依赖 providers 数组最后一项来写 Key，若 addProvider 内部 state 更新异步，getState() 取到的可能是旧数组

**File:** `src/components/Settings/SettingsPanel.tsx:86-92`

**Issue:** Zustand 的 `set()` 是同步的（批处理），但 `addProvider` 里调用了 `storage.set`（同步 localStorage）和 `set({ providers: updated })`。`addProvider` 返回后立即调用 `useProviderStore.getState().providers` 取最新数组是正确的（Zustand set 同步），这里没有异步问题。

但是，逻辑依赖 `updatedProviders[updatedProviders.length - 1]` 来找新建的 Provider 是脆弱的——如果两个事件在同一 microtask 中执行 `addProvider`（理论上不可能，但防御性编程角度），最后一个 Provider 可能不是刚加的那个。

更健壮的做法是让 `addProvider` 返回新建的 id，或者接受 callback：

```ts
// providers.ts addProvider 修改为返回 id：
addProvider(config): string {
  const id = crypto.randomUUID();
  ...
  return id;
}

// SettingsPanel.handleSave：
const newId = addProvider({ name: data.name, baseURL: data.baseURL, model: data.model, isBuiltIn: false });
if (data.apiKey) setKey(newId, data.apiKey);
```

### IN-03: icons.tsx AlertIcon 的 path d 属性含硬编码空格缺陷（`0-3.42 0z` 而非 `0 -3.42 0z`）

**File:** `src/components/icons.tsx:105`

**Issue:** `AlertIcon` 的 path：

```
"M10.29 3.86L1.82 18a2 2 0 0 0 1.71 3h16.94a2 2 0 0 0 1.71-3L13.71 3.86a2 2 0-3.42 0z"
```

其中 `0-3.42` 会被 SVG 解析器视为 `0` 和 `-3.42`（数值连接），在大多数浏览器上可能正确解析，但标准写法应为 `0 -3.42`（加空格）。不同 WebView 版本（尤其是 EdgeHTML 旧版本或 Office 内嵌 WebView）可能解析行为有差异，导致三角形图标变形。

### IN-04: FallbackInsertMenu 的错误处理使用 console.warn 记录错误细节，可能暴露宿主 API 错误信息到开发者工具

**File:** `src/components/ChatBubble.tsx:181`

**Issue:** `console.warn('[Aster] FallbackInsertMenu insert 失败', err)` 将完整的 `err` 对象（包含 `HostApiError.hostError`，即原始 Office.js 错误）打印到控制台。虽然不含 API Key，但 Office.js 错误信息可能包含文档路径、用户名或工作表内容片段，构成隐私泄露风险。与 `chat.ts:294` 的同类处理（`acceptToolCall` 失败）属于相同模式，建议统一改为只记录 error code 而非完整对象：

```ts
console.warn('[Aster] FallbackInsertMenu insert 失败', err instanceof Error ? err.message : String(err));
```
