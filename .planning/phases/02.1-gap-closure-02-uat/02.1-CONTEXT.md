### G-02 — 选区胶囊与 InputBar 对齐（PLAN 02.1-02）

### G-03 — ChatStream 流式不自动滚到底（PLAN 02.1-03）
- **D-10:** 修复 ChatStream 滚动 useEffect 依赖：从 `messages.length` 改为 ① 最后一条 message 的 content 长度，或 ② 订阅 chatStore 的流式 token 追加事件。
- **D-11:** 引入「粘底」策略：① 用户在底部（scrollTop + clientHeight ≥ scrollHeight - 8px）时流式自动滚到底；② 用户主动向上滚阅读历史时**不打断**（识别 scroll up 后 set `stickToBottom=false`，直到用户再滚回底部）。
- **D-12:** UAT 必测：流式输出长回答（≥ 30 行），观察自动滚到底；中途滚到顶看历史，确认不被打断；滚回底部确认重新粘底。

### G-04 — DeepSeek 成本徽章显 ¥（PLAN 02.1-04）
- **D-13:** 排查三步走：① CostBadge 的 `isBuiltIn` 判断要正确识别 `provider.id === 'deepseek'`（不能用 baseURL 判，因为用户改 baseURL 仍是内置）；② chatStore 的 sendMessage 必须正确收集 streamSSE 末尾的 `SSEUsage`（02-02 已做的 `stream_options.include_usage:true` 注入）；③ `calcCostCny` 输入参数正确接 model + token usage，model 命中内置单价表。
- **D-14:** 维持 D-17 边界：内置 DeepSeek / aihubmix 显「N token · ¥X.XXXX」；自定义 Provider 只显「N token」。
- **D-15:** UAT 必测：用 DeepSeek 发一条至少 100 token 输出的消息，气泡下成本徽章必须形如「本次：N token · ¥X.XXXX」（X 非 0）。

### G-05 — AI tool-calling 写文档（D-16 修订）（PLAN 02.1-05）
**这是 02.1 最重的一条改动，跨 Provider / chatStore / Settings / ChatBubble / 三宿主 adapter。**
- **D-16-修订:** 移除 ChatBubble 默认的「插入到文档」按钮（保留 `onSettings` CTA 用于错误深链不变）。
- **D-17:** LLM 调用挂载一个 OpenAI 风格 tool：
  ```
  name: insert_to_document
  description: 把一段文本写回当前 Office 文档（光标处或选中区域）
  parameters: { text: string, position: "cursor" | "replace_selection" | "append_end" }
  ```
  挂载发生在 openai-compat 客户端层，**所有 LLM 请求都带这个 tool**，对模型透明。
- **D-18 Provider 能力探测:** Provider 第一次发请求时探测 tool-call 能力——具体方式：发请求时无条件带 `tools: [...]`，若 Provider 返回 4xx 且错误包含 "tool" / "function" 关键字，则在 providerStore 标记 `provider.supportsToolCall = false`，**当前请求**不重试（让用户看到「该 Provider 不支持 tool-call」的 toast/banner，提示已回退手动菜单），**之后**对此 Provider 的请求不再带 tools。
- **D-19 Settings 开关:** Settings 加一个新的全局选项「AI 自动写文档」，默认 = `confirm`（用户确认）。值：`auto`（直接写）/ `confirm`（预览 + 接受/拒绝）。
- **D-20 confirm 模式 UI:** 当模型在响应中调 `insert_to_document` tool 时，气泡里插入一个「插入预览」卡片：
  - 显示 tool args（text 截断 200 字 + position 中文标签）
  - 「✓ 插入」「✗ 拒绝」两个按钮
  - 用户点击后才真的调 `adapter.insert({type:'text', text, position})`
  - 用户拒绝不需要回报给 LLM（不闭环 tool_result）；接受后也不闭环——本阶段 tool-call 是单向的，不进多轮 tool loop（要进多轮 → Phase 3+）
- **D-21 auto 模式行为:** Settings 切到 `auto` → 模型一调 tool 就直接走 `adapter.insert()`，气泡里只显示「已写入 X 字到 Y 位置」的轻量回执。
- **D-22 不支持 tool-call 的 Provider 回退:** 若 `provider.supportsToolCall === false`，ChatBubble 右上角显示一个「插入 ▾」菜单（光标处 / 替换选区 / 追加末尾三选项），点击调 `adapter.insert()`。**只有这种回退场景才有按钮**，正常 tool-call 路径不出按钮。
- **D-23 adapter.insert 边界:** 三宿主 adapter 的 `insert({type:'text', text, position})` 已在 Phase 02-05 落地最小 text 写回（D-16）。本阶段只新增 `position` 参数路由：cursor / replace_selection / append_end 三种语义；具体 API 调用沿用 Phase 02-05 的实现（PPT textRange.text 赋值、Excel range.values、Word body.insertText）。不做结构化（slides/formula）—— Phase 4-6 范围。
- **D-24 UAT 必测（5 子场景）:** ① confirm 模式下让 DeepSeek 写一段话到 Word → 出预览卡 → 接受 → Word 写入；② confirm 模式 → 拒绝 → Word 无变化；③ auto 模式下让 DeepSeek 写到 PPT 选中 slide → 直接写入 + 显示回执；④ 配一个故意不支持 tool-call 的自定义 Provider（如旧的 OpenAI 兼容代理）→ 发请求 → 看到「该 Provider 不支持 tool-call」提示 + 气泡右上「插入 ▾」按钮；⑤ Excel 三宿主写入 OK。

### G-06 — Settings 编辑 Provider 布局错位（PLAN 02.1-06）
- **D-25:** ProviderForm「保存 / 取消」按钮 = **sticky footer 固定底部**，编辑表单字段在中部可滚动区。
- **D-26 SettingsPanel 分组重构:** 设置页分成三个明确分区，按顺序：
  ① **当前编辑表单**（仅在编辑某 Provider 时才出现，含 sticky 保存/取消）
  ② **Provider 列表**（增删 / 切默认）
  ③ **全局选项**（「AI 自动写文档」开关 D-19、「自动附带选区」开关、「重看引导」链接、隐私说明）
  「全局选项」**绝不能**与「当前编辑表单」混排——这是 G-06 的核心症状。
- **D-27 UAT 必测:** 编辑 DeepSeek Key → 「保存 / 取消」固定在底部可见；底部不串「自动附带选区」/「重看引导」。

### G-07 — KEY_INVALID 被识别成 NETWORK（PLAN 02.1-07）
- **D-28:** 修 `src/lib/openai-compat` 的 `mapHttpError`：在 fetch throw 路径（catch 块）里，**再尝试一次解析**——若 `error instanceof TypeError && error.message.match(/Failed to fetch|NetworkError/)` 且**响应状态码尚未读到**，先 ping `OPTIONS ${baseURL}` 或检查 `navigator.onLine`，区分**真网络断**与**CORS preflight reject / 401 throw**。如果是后者，归类 KEY_INVALID（DeepSeek 401 在某些浏览器路径会以 TypeError 抛出而非 response）。
- **D-29 fallback 策略:** 若 ping 不可行（无后台、cross-origin），按以下信号判 KEY_INVALID：① 当前 request 是已配 Key 的合法 baseURL；② 错误是 TypeError 而非具体 status；③ `navigator.onLine === true`。三条都满足 → 归 KEY_INVALID 并显示「{Provider 名} Key 可能无效，前往设置 →」。注意文案里**不要硬说 100% 是 Key 错**——可能性表述。
- **D-30 UAT 必测:** 把 DeepSeek Key 改成 `sk-invalid` 发消息 → 错误气泡显示「DeepSeek Key 可能无效，前往设置 →」（KEY_INVALID 文案，不能是 NETWORK 文案）；同时验证「真网络断」（断 WiFi）仍正确显示 NETWORK 文案。两类必须分开。

### G-08 — SelectionPill 眼睛 toggle（D-15 修订）（PLAN 02.1-08）
- **D-31 修订 D-15:** 选区胶囊默认仍自动附带（保留 D-15 默认行为），但右侧加眼睛图标 toggle：眼睛开 = 附带 / 眼睛闭 = 不附带但胶囊仍在。
- **D-32 持久化:** toggle 状态持久化到 partitioned localStorage，key = `selection_attach_enabled`（默认 true）。**不**走 chatStore（chatStore 是内存级），用 providerStore 或新建小 uiStore。
- **D-33 × 行为保留:** 胶囊右侧 × 仍然存在，但行为收窄为「本次会话隐藏胶囊」（不持久化）；下次打开 Task Pane 或刷新页面胶囊恢复。
- **D-34 上传 / send 取值:** InputBar `handleSend` 调 `adapter.getSelection()` 时，先看 `selection_attach_enabled`——false 则 `selection = undefined`；true 才真的取选区。
- **D-35 UAT 必测:** ① 默认眼睛开 → 选 slide 发消息 → AI 看到「在第 N 张 slide」；② 点眼睛切闭 → 同样发 → AI 不再看到选区；③ 刷新页面 → 眼睛仍是闭的（持久化生效）；④ × 关掉胶囊 → 刷新 → 胶囊回来（× 不持久化）。
