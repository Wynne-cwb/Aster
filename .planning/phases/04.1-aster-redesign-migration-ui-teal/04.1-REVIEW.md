## Critical Issues

### CR-01: CIRCUIT_OPEN「重新试试」把 `undefined` 当 adapter 传给 runAgent，点击必崩

> ✅ **RESOLVED 2026-05-29（`53fa6eb`，已 push 部署）** — `ToolResultCard` 改用 `useAdapter()` 取真实 adapter（与 `ChatBubble` retry 同源），并新增守门测试锁住第 3 参 = 真实 adapter，防回归。源头为 phase 04（`7ca7c9a` ERR-04），非 04.1 引入。

**File:** `src/components/ChatStream.tsx:153`
**Issue:**
`runAgent` 的签名（`src/agent/agentStore.ts:58-62`）要求 `adapter: DocumentAdapter` 为**非可选**参数。这里却写：

```tsx
onClick={() => { void store.runAgent(prompt, undefined, undefined as never); }}
```

`undefined as never` 强制绕过类型检查，把 `undefined` 当 adapter 注入。`runAgent` → `runAgentLoop`（`src/agent/loop.ts:49`）在循环一开始就执行 `const host = adapter.capabilities().host;`，对 `undefined` 调用会同步抛 `TypeError: Cannot read properties of undefined (reading 'capabilities')`。

后果：用户在熔断（连续 3 次工具失败）后点「重新试试」必然 crash agent run，重试功能完全不可用——而这正是熔断红卡上唯一的恢复入口。`undefined as never` 强转把本应是编译期错误的 bug 藏到了运行期。

注：CIRCUIT_OPEN 分支在本阶段被重皮成 `err-bubble` 形态（D-06），按钮就在改动范围内，无论 bug 何时引入都应在此修复。

**Fix:**
ToolResultCard 拿不到 `adapter`（它在子组件内、未走 AdapterContext）。两种修法：

1) 让 ChatStream 把已有的 `adapter`（顶层 `const adapter = useAdapter()`）透传进 ToolResultCard，再传给 `runAgent`：

```tsx
function ToolResultCard({ message, adapter }: { message: Message; adapter: DocumentAdapter }): ReactElement {
  // ...
  onClick={() => { void store.runAgent(prompt, undefined, adapter); }}
```

2) 或复用 chatStore 既有的 `retryMessage(messageId, adapter)` thin delegate（已在 ChatStream 用 `useAdapter()` 注入）。无论哪种，都必须删掉 `undefined as never`——它是这个 bug 能逃过 `tsc --strict` 的唯一原因。

## Warnings

### WR-01: 未配 Key 提示条不会在配置 Key 后自动消失（banner 反应性失效）

> ✅ **RESOLVED 2026-05-29（`7821c75`，已 push 部署）** — providerStore 新增响应式 `configuredKeyIds`（仅 id、不含 Key 值，遵守 T-02-18），`setKey`/`removeProvider`/`hydrate` 维护；App `hasKey` selector 改读它。+3 守门测试。根因自 phase 02（`69c3b32`/`6ab6d08`），04.1 重皮继承，非新引入。

**File:** `src/App.tsx:43`
**Issue:**
```tsx
const hasKey = useProviderStore((s) => !!s.getKey(s.defaultLLMProviderId));
```
`getKey`（`src/store/providers.ts:136-138`）读的是 `storage`（localStorage），**不是 Zustand state**；而 `setKey`（同文件 132-134）只写 storage、从不调 `set(...)`。因此当用户在 Onboarding 或 Settings 里填好 Key（`Step1Keys.handleNext` → `setKey('deepseek', dsKey)`），providerStore 不产生任何 state 变更 → App 的 `hasKey` selector 不会重新求值 → 顶部「请先配置 API Key」红条不会消失，直到某个无关的 store 变更恰好触发重渲染。这是重皮 D-01 引入 pane-banner 后的功能回归。

**Fix:**
让 Key 进入响应式状态。最小改动：在 providerStore 维护一个 `keyVersion`（或 `configuredKeyIds: string[]`）字段，`setKey` 末尾 `set((s) => ({ keyVersion: s.keyVersion + 1 }))`；App 订阅该字段以驱动重算。或将 banner 显隐改由订阅 `keyVersion` 的派生 selector 计算。

### WR-02: Send 按钮 busy 时 disabled，却标 `aria-label="停止"` + 显示 StopIcon（误导性可达性标签）

**File:** `src/components/InputBar.tsx:116-125`
**Issue:**
```tsx
<button
  className="send-btn"
  data-streaming={isAgentBusy || undefined}
  disabled={isAgentBusy || !text.trim()}
  aria-label={isAgentBusy ? t`停止` : t`发送`}
>
  {isAgentBusy ? <StopIcon size={11} /> : <SendIcon size={13} />}
</button>
```
busy 时按钮同时 `disabled` 且 `aria-label="停止"`、显示停止图标。屏幕阅读器会朗读「停止 按钮 不可用」，视觉上也呈现一个停止键——但它点不动（真正的中止在 AgentControlBar）。这给用户/辅助技术一个「这里能停」的错误暗示，实际不能。CSS `.send-btn[data-streaming]` 还专门给了高亮态，强化了「可点」的错觉。

**Fix:**
busy 态要么让它真正成为 stop 入口（`disabled={!text.trim() && !isAgentBusy}`，`onClick` busy 时调 `useChatStore.getState().stopStreaming()`），要么 busy 时就标成发送的禁用态而非停止：

```tsx
disabled={isAgentBusy || !text.trim()}
aria-label={t`发送`}
// busy 时不显示 StopIcon、不加 data-streaming 高亮，避免「能停」误导
```
推荐前者（让停止键名副其实），与 `data-streaming` 高亮的视觉意图一致。

### WR-03: CIRCUIT_OPEN 重试丢失原始 selectionCtx

**File:** `src/components/ChatStream.tsx:153`
**Issue:**
即便修好 CR-01 的 adapter，`runAgent(prompt, undefined, ...)` 的第二参 `selectionCtx` 仍写死 `undefined`。原始请求若依赖选区内容（这是 Aster 的核心用法），重试会在**无选区上下文**下重跑，行为与首次不一致，可能再次失败或产出错误结果。

**Fix:**
重试应复用首次的取选区逻辑（与 `InputBar.handleSend` 一致）：读 `attachEnabled`，为 true 时 `await adapter.getSelection()` 再传入。最简单是直接走 `chatStore.retryMessage` 路径，让取选区逻辑集中一处，避免在红卡里重复实现。

### WR-04: AlertIcon 的 SVG arc path 缺 flag 参数，渲染脆弱

**File:** `src/components/icons.tsx:105`
**Issue:**
```tsx
<path d="M10.29 3.86L1.82 18a2 2 0 0 0 1.71 3h16.94a2 2 0 0 0 1.71-3L13.71 3.86a2 2 0 0-3.42 0z" />
```
末尾 arc 段 `a2 2 0 0-3.42 0z` 的参数个数与 Lucide 原版 `triangle-alert`（`...a2 2 0 0 0-3.42 0z`，三个 `0`：rotation/large-arc/sweep）不一致——这里只有 `0 0` 两个 flag 位后接 `-3.42`。依赖浏览器对「flag 紧凑写法」的容错才可能正确解析成 rotation=0/large-arc=0/sweep=0/dx=-3.42。不同 SVG 解析器对 flag 是否允许与后续数值连写处理不一，存在三角形顶点闭合错误、图标变形的风险（错误气泡、pane-banner 都用此图标，属高频可见元素）。

**Fix:**
补回缺失的 flag、用空格分隔到 7 参齐全，对齐 Lucide 原版：

```tsx
<path d="M10.29 3.86 1.82 18a2 2 0 0 0 1.71 3h16.94a2 2 0 0 0 1.71-3L13.71 3.86a2 2 0 0 0-3.42 0z" />
```

### WR-05: PaperclipIcon 的 `size` 标为可选却无默认值

**File:** `src/components/icons.tsx:194`
**Issue:**
```tsx
export function PaperclipIcon({ size = 24 }: { size?: number }): ReactElement {
```
等等——实测此处**有**默认值 `size = 24`，但函数签名与同文件其他图标（如 `ChevronDownIcon({ size = 24, className }: { size?: number; className?: string })`）相比缺少形参默认对象 `= {}`。`PaperclipIcon` 当前唯一调用点 `InputBar.tsx:113` 传了 `size={15}`，所以不触发；但若将来无参调用 `<PaperclipIcon />`，因解构对象自身没有 `= {}` 兜底，会抛 `Cannot destructure property 'size' of undefined`（与 `SendIcon`/`AlertIcon` 等带 `= {}` 的写法不一致）。属一致性/健壮性隐患。

**Fix:**
```tsx
export function PaperclipIcon({ size = 24 }: { size?: number } = {}): ReactElement {
```
同理核对 `ChevronIcon` / `ChevronLeftIcon` / `AlertCircleIcon` / `DocumentIcon` / `ChevronDownIcon`：凡可能被无参调用的，统一补 `= {}`。

### IN-04: ExpandedBody 的 JSON 兜底会渲染字面量 "undefined"

**File:** `src/components/ChatStream.tsx:73`
**Issue:** `return <pre className="wb-action-body">{JSON.stringify(result, null, 2)}</pre>;` 当 `result === undefined` 时 `JSON.stringify(undefined)` 返回 JS `undefined`，React 渲染为空——尚可；但若 `result` 是 `{ ok:true, data:undefined }` 等部分结构，pre 块会显示对用户无意义的原始 JSON。展开区本是兜底路径，呈现技术 JSON 给中文职场用户体验欠佳。
**Fix:** `result` 为空时给一句人话占位（如「无更多详情」），而非裸 JSON / 空白。

### IN-06: empty-state 用 `.chat-scroll` 容器，正常消息列表用 `.aster-messages`（两套滚动容器不一致）

**File:** `src/components/ChatStream.tsx:297 vs 347`
**Issue:** 空态返回 `<div className="chat-scroll">`，有消息时返回 `<div className="aster-messages">`，两者是不同的 CSS 滚动容器（`.chat-scroll` 与 `.aster-messages` 各有独立 flex/overflow 规则）。功能无害，但同一区域用两套容器类名增加维护成本，且从空态切到有消息时整个滚动容器节点类型变化（React 会卸载重建）。
**Fix:** 统一用同一外层滚动容器类，空/非空仅切换内部内容，减少类名与重建开销。
