## Spacing Scale

This phase introduces NO new spacing tokens. It reuses the LOCKED `--space-1..6` scale shipped and UAT-passed in Phase 04.1 (2026-05-29). All values are multiples of 4.

| CSS Token | Value | Phase 6 usage |
|-----------|-------|---------------|
| `--space-1` | 4px | Icon-label gap inside chip, badge padding |
| `--space-2` | 8px | Chip horizontal padding, modal-foot button gap |
| `--space-3` | 12px | Modal internal section gap (e.g. host-card padding — reused from settings-and-onboarding pattern) |
| `--space-4` | 16px | `.empty` section gap between chips, modal-foot horizontal padding |
| `--space-5` | 20px | `.modal` internal padding (reused from shipped modal) |
| `--space-6` | 24px | `.empty` outer vertical padding |

**Exceptions:** None new this phase.

**Note on pre-existing component internals:** `.empty` gap of 14px and `.modal` padding of 20px are LOCKED shipped values from Phase 04.1 — reused unchanged, not new declarations. The value 14px (gap inside `.empty`) is a deliberate, shipped, UAT-passed exception to the space scale, used for the empty-state vertical rhythm. The value 12px (--space-3) is a multiple of 4 and is part of the locked scale.

Touch targets: any interactive chip must have a minimum height of 36px (consistent with shipped `.btn` and `.btn-sm` sizing). No new 44px exception required; the existing `.btn` sizing already meets reachability for the Task Pane at 350px fixed width.

## Color

Single-accent teal system. 60/30/10 split is intentional — this is a deliberate quiet/克制 design language, not a gap.

| Role | Light token + value | Dark token + value | Ratio | Usage |
|------|--------------------|--------------------|-------|-------|
| Dominant surface | `--bg` / `--surface` = `#FFFFFF` | `--bg` / `--surface` = `#0E0E10` | 60% | Pane background, modal surface, empty-state background |
| Secondary | `--surface-2` = `#F3F2EE` / `--bubble-ai-bg` = `#EEEEF0` | `--surface-2` = `#1F1F21` / `--bubble-ai-bg` = `#1F1F23` | 30% | Chip ghost hover bg, modal-scrim overlay, `.empty` container bg (if applicable), host-card bg |
| Accent (teal) | `--accent` = `#009887` / `--accent-hover` = `#007A6E` | `--accent` = `#4FC9B8` / `--accent-hover` = `#6BDBC9` | 10% | See list below |
| Destructive | `--error` = `#DC2626` | `--error` = `#F87171` | — | Error states only — no NEW destructive UI this phase |

**Accent (`--accent`) reserved for — explicit list:**
1. `.btn-primary` background (modal「开始使用」CTA)
2. Chip trailing `arrowRight` icon on hover (`color: var(--accent)` or driven by `--text-3` → accent transition)
3. Focus ring (`--ring-focus` = `0 0 0 2px var(--bg), 0 0 0 4px var(--accent)`) on all focusable chip/button elements
4. `.host-icon` color in retained host-card (if referenced from existing onboarding — reused only)
5. User message bubble background (`.bubble-user`, reused, not new)

**Destructive note:** No NEW destructive confirmation UI in Phase 6. The diff-log undo confirmation shipped in Phase 5 is reused unchanged. The `--error` token is documented here for completeness.

**Both themes must be covered.** Every new class must declare behavior under both `[data-theme="light"]` and `[data-theme="dark"]`. The shipped token blocks in `src/styles.css` already define both; new UI only references existing variables.

**Anti-pattern enforcement:** Never write `#009887` or `#4FC9B8` directly. Always use `var(--accent)`. Never write `#FAFAF8` (design-spec warm-white — UAT overridden to pure white `#FFFFFF`). No multi-color gradients. No `backdrop-filter`.

## Component Contracts — Phase 6 New UI

### 1. Empty-State Killer-Scenario Chips

**Location:** `src/components/ChatStream.tsx` — the `// D-03：不渲染 chips，等 Phase 6` hook block inside the `messages.length === 0` branch.

**Pattern:** Reuse existing `.suggestions` + `.btn .btn-ghost` ghost-button chip pattern from Phase 04.1 (described in `chat-and-bubbles.md` §空状态). No new visual language.

**Structure:**
```
.empty
  ├ .empty-mark (logo pulse, already shipped)
  ├ h3 (already shipped)
  ├ p  (copy updated — see Copywriting)
  └ .suggestions          ← Phase 6 fills this
      ├ button.btn.btn-ghost.btn-sm  (PPT chip 1)
      ├ button.btn.btn-ghost.btn-sm  (PPT chip 2)
      └ button.btn.btn-ghost.btn-sm  (PPT chip 3)
      (Excel/Word: 2-3 chips respectively)
```

**Visual hierarchy / focal point (empty state):** Landing gaze order is `.empty-mark` (logo pulse, primary focal point on entry) → `h3` heading (16px semibold) → `p` body (13px `--text-2`) → `.suggestions` chips. The chips are the primary *interactive* target (they are what the user acts on), but the logo pulse remains the entry focal anchor — chips draw the eye via teal arrow on hover, not via standing color weight (they are ghost buttons, deliberately quiet until hovered/focused).

**Each chip anatomy:** `[leftIcon 14px stroke 1.5] [label 13px left-aligned] [arrowRight 13px --text-3 trailing]`

**Interaction:**
- Click → FILL `InputBar` with the chip's seed-prompt string. Do NOT auto-send / auto-submit. User reads the prefilled text, may edit, then presses Send.
- Hover → chip slides right 2px (`transform: translateX(2px)`, `transition: transform var(--dur-fast) var(--ease-out)`). Trailing arrowRight shifts with it (no separate transform needed — it's inside the same button).
- Focus → `box-shadow: var(--ring-focus)`.
- Host detection → `useAdapter().capabilities().host` determines which chip set to render. Never show cross-host chips.

**Host-specific chip copy (locked by D-14 killer scenarios):**

PowerPoint (3 chips):
1. `「帮我做一份 Q3 销售复盘 PPT，给 leadership 看，重点华东」`
2. `「把左下角那张图加红色边框，再往右移 10 px」`
3. `「在最长的那页后面补一页总结要点」`

Excel (3 chips):
1. `「帮我清洗这份数据、加公式、画个图，再给三句话洞察」`
2. `「看看哪个产品卖得最好，做个对比图」`
3. `「检查一下有没有重复行，帮我去掉」`

Word (3 chips):
1. `「帮我把整篇文档润色一遍，口语改成正式书面」`
2. `「把我选中的这段改得更正式一点」`
3. `「帮我生成一个文档摘要，三句话以内」`

**Bundle note:** Chip copy is plain string literals (zero bundle impact). The chip component logic is minimal and lives in the main chunk (no lazy load needed). CI gate: initial `main-*.js` must remain ≤ 82KB gzip.

**Honest disabled:** No chips are in a disabled state. If the host is not recognized, render no chips (safe fallback to empty `.suggestions` or omit the block entirely).

### 2. Single-Step Onboarding

**Decision:** D-18 / D-19 / D-21 — Collapse 2-step modal to 1 step. Delete `Step2Guide.tsx` entirely. Step1Keys becomes the sole and final step.

**What changes in `OnboardingModal.tsx`:**
- Remove `step` state (`useState<1 | 2>(1)` → no longer needed)
- Remove `goNext()` / `goBack()` handlers
- Remove `Step2Guide` import and render
- Keep `.modal-scrim` / `.modal` / `.modal-brand` structure unchanged
- Remove `{step === 1 ? '01' : '02'} / 02` step counter — replace with nothing (single step, no counter)
- `onComplete` is now called directly by Step1Keys (rename `onNext` prop to `onComplete`)

**What changes in `Step1Keys.tsx`:**
- Rename prop `onNext: () => void` → `onComplete: () => void`
- Write `storage.set(STORAGE_KEYS.ONBOARDING_SEEN, true)` before calling `onComplete()` (previously Step2Guide wrote this)
- Update `.modal-sub` copy (see Copywriting)
- Change primary CTA from 「下一步」→「开始使用」
- The `handleNext` function becomes `handleComplete`

**What is deleted:** `src/components/Onboarding/Step2Guide.tsx` — entire file deleted. No references may remain.

**Step1→main transition:** After Step1 completes, `onComplete()` triggers the parent App's `setShowOnboarding(false)` (or equivalent). D-21 requires verifying this path works correctly in the single-step flow. The existing 2-step transition bug is expected to dissolve with the simpler path, but must be confirmed in UAT.

**`.modal-brand` step counter:** Remove `<span className="brand-step">01 / 02</span>` (or the equivalent string). No replacement — single step has no counter.

**Layout:** `.modal` keeps all existing structure (`max-width: 320px`, `--radius-4`, `--shadow-pop`, `padding: 20px`, `gap: 14px`). No new layout work.

### 4. ONB-02 Tool-Step Summaries (satisfied by existing system)

**Status:** Satisfied by the existing mandatory `humanLabel` system (Phase 3 lint gate + Phase 5 enforced). Every tool must export `humanLabel(args) => string` (Chinese human-readable). Phase 6's new write tools must follow the same requirement — non-compliance causes TS/lint compile failure.

**New UI work:** None. The `humanLabel` strings are copywriting output in the tool definition files, not UI components.

**Copywriting contract for humanLabel (examples, to guide executors):**
- `set_shape_property` → `「将第 N 张幻灯片的形状『XXX』边框颜色改为红色」`
- `move_shape` → `「将形状『XXX』向右移动 10 px」`
- `apply_formula` → `「在 B2 单元格写入公式 =SUM(A2:A10)」`
- `insert_chart` → `「在当前工作表插入柱状图」`

- Pattern: `「动词 + 对象描述 + 具体参数」`，全中文，无 raw tool name，无英文参数键名。
